Architecture Overview

Four pillars of the PWA offline system — how they connect

System Architecture
📱 PWA Client (iOS + Android + Desktop)
Dexie.js + Encryption Middleware
IndexedDB tables: territories, addresses, visits, assignments, meetings, meetingPoints, fieldGroups
AES-256-GCM encryption on all tables
Sync Engine
↓ Pull: GET /sync/pull?since=
↑ Push: POST /sync/push (queued changes)
⚡ Conflict: version mismatch → resolution UI
Device Manager
UUID + user-agent + screen + platform
POST /devices/register on first login
Checks revocation status on every sync
Offline Queue
Mutations stored in "pendingChanges" table
Auto-push on navigator.onLine → true
Manual "Sync now" button
🖥️ hub-api (Fastify + Prisma)
Sync Endpoints
GET /sync/pull — delta since timestamp
POST /sync/push — accept client changes
GET /sync/status — last sync metadata
Device Registry
POST /devices/register
GET /devices — admin list all
DELETE /devices/:id — admin revoke
GET /devices/me — client check status
Conflict Resolution
version column on all syncable tables
Push rejected if client version < server version
Returns both versions for client-side merge UI
Encryption Key Endpoint
POST /devices/encryption-key
Derives AES key from OIDC token + device ID
Key cached client-side in memory only
Data Flow
Login: Keycloak OIDC → token → POST /devices/register → GET encryption key → initial pull → encrypt & store in Dexie
Online: API calls as normal + background sync to keep Dexie fresh
Offline: Read from Dexie (decrypt) → mutations queued in pendingChanges → auto-push on reconnect
Revoked: GET /devices/me returns revoked → wipe Dexie → show reason → force re-login
