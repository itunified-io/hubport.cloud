Territory Polygon Editing: How should it work?

Choose the editing interaction model for moving vertices and adjusting boundaries

A

Inline Map Editing (Recommended)

Edit directly on the existing territory detail map. Click "Edit" → vertices become draggable handles on the map. Drag to move, click edge midpoints to add vertices, right-click to delete. Snap-to-road button snaps all vertices to nearest OSM road/street. Save/Cancel buttons appear.

Pros

  • No page navigation — stays in context
  • See addresses, assignments while editing
  • Familiar Google Maps-style UX
  • Works for both new and existing territories

Cons

  • More complex map interaction code
  • Small screen real estate on mobile
B

Full-Screen Editor Page

Navigate to dedicated /territories/:id/edit page with full-screen map. Sidebar with vertex list, snap controls, undo/redo stack. More room for tools but loses context of the detail page.

Pros

  • Maximum map space
  • Room for advanced tools (undo stack, vertex table)
  • Clean separation of view vs edit

Cons

  • Context switch — loses address/assignment view
  • Extra route + page component to build
  • More navigation complexity

Snap-to-Road Feature (included in both)

When editing, a "Snap to Roads" button would query the existing /territories/snap-context endpoint (already fetches OSM roads, buildings, water in bbox). Each vertex snaps to the nearest road geometry within a configurable threshold (e.g., 50m). User sees a preview before confirming.
