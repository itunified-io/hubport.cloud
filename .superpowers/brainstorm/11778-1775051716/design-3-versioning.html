2. Restore Flow

What happens when user clicks "restore ↩"
1 User clicks "restore ↩" on v1
2 Client calls POST /territories/:id/restore { versionId }
3 Server loads v1 boundaries → runs auto-fix pipeline on them (they may now violate current neighbors)
4 If modified → preview dialog (same as edit preview)
5 Accept → saves boundary + creates v4 (changeType: "restore", summary: "Restored from v1")

Non-destructive: Restore never overwrites history. It creates a new version pointing back to the restored source. The full version chain is always preserved. Even after restore, all previous versions remain accessible.

3. When Versions Are Created

creation
New territory gets its first boundary via CreationFlow → v1
manual_edit
User edits boundary via vertex handles → new version on save
auto_clip
Auto-fix pipeline modifies geometry (congregation clip or neighbor clip) → new version
import
KML/CSV import sets boundary → v1 (or new version if boundary already existed)
restore
User restores a previous version → new version with reference to source
