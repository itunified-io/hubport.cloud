Design Section 4/4 — Migration Plan for Existing Territories

How to handle territories that already exist in the DB when this feature ships.

1. Bootstrap v1 Snapshots

One-time migration — runs on first deploy
1 Prisma migration adds TerritoryBoundaryVersion table
2 Seed script runs: for each Territory WHERE boundaries IS NOT NULL
3 Create version record: { version: 1, changeType: "import", changeSummary: "Initial snapshot from existing boundary" }
4 Idempotent — skips territories that already have version records

When: This runs as part of the Prisma seed script (prisma/seed.ts), which executes on db push / container restart. It's idempotent — safe to re-run. Existing boundaries are NOT modified, just snapshotted as v1.

2. Initial Violation Detection

What happens on first map load after deploy
1 User opens Kartenansicht (map view)
2 GET /territories/violations scans all territory boundaries
! Existing violations appear as warning badges on the map
✓ User fixes each one manually: click badge → detail page → Edit → auto-fix preview → save

No auto-fix on deploy: Existing boundaries are NOT automatically clipped during migration. The user sees violations and fixes them one-by-one via the normal edit flow. This preserves intentional boundary decisions and gives the user control over each change.

3. No Breaking Changes

Concern Impact
Existing GET /territories ✅ Unchanged — still returns boundaries in same JSON shape
Existing PUT /territories/:id ✅ Enhanced — now runs auto-fix pipeline, but response is backward-compatible (adds autoFix field)
Territories without boundaries ✅ No version record created — they get v1 when boundary is first drawn
Congregation boundary territory ✅ Excluded from violations check (it IS the boundary)
KML import ✅ Import now runs auto-fix pipeline — imported boundaries get clipped automatically

4. Seed Role Migration

// seed-roles.ts already runs on every deploy (idempotent upsert)
// New "Territory Servant" role gets created automatically
// FLAG_TO_APP_ROLE picks it up on next login/token refresh
Existing users with territory_servant flag:
→ Next login: flag auto-maps to new "Territory Servant" AppRole
→ Immediately get territory edit permissions
→ No manual action needed by admin
